# Архитектура Copilot 1CO

## Слои фронтенда (`site/`)

- **UI-компоненты (`site/js/components`)**
    - Чистые визуальные элементы и небольшие интерактивные блоки без доступа к API или хранилищу.
    - Принимают данные и коллбеки через параметры или зависимости, не знают о формате запросов и расположении данных.

- **Фичи (`site/js/features`)**
    - Сценарии использования и экраны (окна, модалки, сложные взаимодействия).
    - Собирают UI-компоненты и обращаются только к сервисам, а не напрямую к API/DB/`window`.

- **Сервисы (`site/js/services`)**
    - Доменная логика и работа с внешними интерфейсами.
    - Включают:
        - `apiClient` / HTTP-клиент.
        - `revocationService` и др. специализированные сервисы.
        - Сервис уведомлений, экспорта данных и т.п.
    - Не зависят от конкретной разметки, только от конфигов и утилит.

- **Инфраструктура и утилиты**
    - `site/js/utils`: форматирование, парсинг, валидация, вспомогательные функции без побочных эффектов.
    - `site/js/db`: доступ к локальному хранилищу (IndexedDB, localStorage) через единый API.
    - `site/js/ui`: низкоуровневые UI-хелперы (overlay, hotkeys, загрузка шаблонов, системные модалки).
    - `site/js/config.js` и `site/js/constants.js`: единый источник правды для конфигурации и констант.

- **Инициализация приложения**
    - `site/js/app.js`: точка сборки модульной части, инициализация БД, сервисов и ключевых компонентов.
    - `site/js/entry.js`: связывает существующую глобальную логику (`script.js`) с модульной архитектурой через `setDependencies`.

## Слои backend / API

- **Yandex Cloud Function (`yandex-function/crl-checker/index.js`)**
    - HTTP-слой:
        - Маршруты `/api/revocation/check` (GET/POST) и `/api/health`/`/`.
        - CORS, валидация базовых параметров, карты статусов и ошибок.
    - Доменный слой проверки отзыва:
        - Функция `checkRevocation(serial, listUrl)` как единая точка входа.
        - Нормализация серийного номера, выбор стратегии чтения списка отзыва (JSON, текст, CRL).
    - Парсеры и адаптеры:
        - `parseCrlRevokedSerials` и ASN.1-парсер DER/PEM.
        - Вспомогательные функции работы с буферами, датами, hex-представлением.

- **Локальный dev backend (`api/revocation/check.js`, опционально)**
    - Используется как локальная/тестовая реализация того же контракта.
    - Для production-окружения целевым backend является `yandex-function/crl-checker/index.js`.

## Контракты и конфигурация

- **Единый контракт ревокационного API**
    - Описывается в отдельном документе `docs/plans/revocation-api-contract.md`.
    - Фронтенд обращается только через сервис `revocationService`, не используя «сырые» URL.

- **Конфигурация сред**
    - Базовые URL и флаги среды задаются через:
        - `REVOCATION_API_BASE_URL` в `site/js/config.js`.
        - invoke URL функции в Yandex Cloud Functions.
        - переменные окружения функции (`REVOCATION_PROXY_URL`, `REVOCATION_LOCAL_HELPER_BASE_URL`) при необходимости.

## Принципы унификации

- Каждая ответственность имеет один источник истины:
    - Одна реализация для каждой бизнес-операции (например, проверка отзыва, операции с избранным и закладками).
    - Одна точка доступа к локальному хранилищу на тип сущности.
    - Единый набор UI-компонентов для однотипных окон (модалки, карточки, списки).
- Фичи не обращаются напрямую к `window`, `localStorage`, `fetch` и т.п.; всё проходит через сервисы и слой инфраструктуры.
- Backend и фронтенд согласованы по контрактам, форматам ошибок и health-check’ам.

## Рекомендуемая миграция к унифицированной архитектуре

1. **Поддержка текущего функционала**
    - Сохранять совместимость мостом `script.js` ↔ `site/js/app.js`/`entry.js` до полной миграции.
    - Новую логику добавлять только в модули (`features/services/utils/db/ui`), минимизируя рост `script.js`.
2. **Инкрементальный вынос логики**
    - Для каждой крупной области (алгоритмы, закладки, внешние ссылки, регламенты, FNS-проверка):
        - Выделить сервисы, работу с БД и утилиты в соответствующие модули.
        - Оставить в `script.js` только тонкий слой привязки к DOM и глобальным переменным, постепенно сокращая его.
3. **Унификация API и ошибок**
    - Использовать `HttpClient` и специализированные сервисы (например, `RevocationService`) для всех HTTP-запросов.
    - Для отображения ошибок применять единый `NotificationService` + маппинг (`utils/error-mapping.js`).
4. **Стандартизация UI-окон**
    - Для модалок использовать:
        - Конфиги из `config.js` (`FULLSCREEN_MODAL_CONFIGS`, `UNIFIED_FULLSCREEN_MODAL_CLASSES`).
        - Общий модальный менеджер (`ui/modals-manager.js`, `utils/modal.js`, `ui/modal-overlay-handler.js`, `ui/unsaved-confirm-modal.js`).
    - Новые окна строить поверх этих механизмов, а старые постепенно адаптировать.
5. **Контроль качества**
    - При изменениях:
        - Прогонять линтер (`npm run lint`) и форматирование (`npm run format`).
        - Поддерживать и расширять тесты (`npm test`) для ключевых доменных модулей (`yandex-function`, `site/js/utils`/`services`).
